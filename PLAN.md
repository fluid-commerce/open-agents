Summary: Replace the web chat path's in-request `webAgent.stream(...)` execution plus `resumable-stream` replay with the reference workflow pattern: start a durable workflow, stream through Workflow DevKit, and let `WorkflowChatTransport` handle reconnects. Keep `packages/agent` as the current `ToolLoopAgent`, add a durable execution mode that limits the top-level agent to one step per workflow step, and reuse `chats.activeStreamId` as the active workflow run id so the existing chat/session model stays intact.

Context: The current web flow in `apps/web/app/api/chat/route.ts` does everything inline: auth, sandbox connection, model resolution, skill discovery, stop-signal subscription, direct `webAgent.stream(...)`, resumable stream registration, and post-finish persistence/automation. Stream resume is implemented with Redis-backed `resumable-stream` in `apps/web/lib/resumable-stream-context.ts` and `apps/web/app/api/chat/[chatId]/stream/route.ts`, while stop uses Redis pub/sub in `apps/web/lib/stop-signal.ts` and `apps/web/app/api/chat/[chatId]/stop/route.ts`. The reference app instead moves orchestration into `src/workflows/run-agent.ts`, streams via `getWritable()`, resumes via `WorkflowChatTransport`, and cancels by `getRun(...).cancel()`. Open Harness already has Workflow wired into Next in `apps/web/next.config.ts` and `apps/web/tsconfig.json`, so the migration is mostly about replacing the chat transport/execution path, not bootstrapping Workflow itself. No schema change is strictly required if `chats.activeStreamId` is repurposed from resumable-stream tokens to workflow run ids; rollout just needs to treat legacy non-`wrun_...` values as stale and clear them.

Approach: Mirror the reference project's workflow loop for the web app, not the CLI. The POST chat route should become a thin "validate + persist latest submitted messages + start workflow + return stream" entrypoint. A new workflow file should own the long-running loop: emit one `start` chunk, run one top-level `openHarnessAgent` step inside a Workflow step, append returned `response.messages` into the next iteration, and finish only when the agent stops or the run is cancelled. Repo-specific work stays around that core: reconstruct sandbox/model/skills inside the workflow step from serializable IDs, keep `activeStreamId` as the authoritative active workflow run id, preserve current assistant-message/sandbox/usage persistence semantics at workflow finalization, and replace request-bound helpers that depend on `getServerSession()` or cookies with explicit `userId`/`sessionId` server helpers so secrets are not serialized into workflow state.
